# OfficeOps Hub 요구사항 정의서

## 1. 문서 목적

이 문서는 `OfficeOps Hub` 프로젝트의 개발 범위, 사용자 역할, 필수 요구사항, 제외 범위를 정의한다.

구현 단계에서 기능 범위가 불필요하게 커지는 것을 방지하고, 백엔드/프론트엔드 개발 기준을 맞추는 것을 목적으로 한다.

## 2. 프로젝트 개요

`OfficeOps Hub`는 회사 내부에서 발생하는 비품 요청, 회의실 예약, 방문 신청, 자산 대여/반납, 근태 관리, 전자결재 업무를 통합 관리하는 사내 운영/HR 관리 시스템이다.

일반 직원은 요청, 예약, 근태 신청, 전자결재 문서를 등록하고 처리 상태를 확인할 수 있다. 팀장은 소속 팀원의 운영 요청과 전자결재 문서를 1차 승인하거나 반려할 수 있고, 운영 담당자는 자산/예약/시설 관련 요청을 최종 승인하고 처리한다. HR 담당자는 연차, 근태 정책, 증명서 발급을 관리하며, 재무 담당자는 지출결의서와 법인카드 사용내역서를 검토하고 처리한다. 관리자는 사용자, 자산, 예약, 자원, 근태, 전자결재, 공통 코드, 감사 이력을 포함한 전체 현황을 관리한다.

## 3. 프로젝트 목표

### 3.1 서비스 목표

- 사내 요청, 예약, 자산 관리 업무를 하나의 시스템에서 처리한다.
- 근태, 연차, 전자결재 문서 신청을 하나의 포털에서 처리한다.
- 직원은 본인의 요청과 예약 상태를 쉽게 확인할 수 있다.
- 직원은 본인의 연차 잔여일수, 연차 사용 내역, 출퇴근 기록, 전자결재 진행 상태를 확인할 수 있다.
- 팀장은 소속 팀원의 요청을 1차 검토할 수 있다.
- 운영 담당자는 요청을 최종 승인하고 담당자, 처리 예정일, 마감일을 관리할 수 있다.
- HR 담당자는 연차 발생/조정/사용 내역, 근태 현황, 증명서 발급 요청을 관리할 수 있다.
- 재무 담당자는 지출결의서와 법인카드 사용내역서를 검토하고 처리할 수 있다.
- 관리자는 전체 요청, 예약, 자산 현황을 한 화면에서 관리할 수 있다.
- 모든 요청 상태 변경 이력을 저장해 처리 과정을 추적할 수 있다.
- 회의실 예약 시 시간 중복을 방지한다.

### 3.2 개발 목표

- Java 21, Spring Boot 3 기반 REST API를 구현한다.
- Spring Security와 JWT 기반 인증/인가를 구현한다.
- React 기반 사용자/관리자 화면을 분리해 구현한다.
- PostgreSQL 기반 관계형 데이터 모델을 설계한다.
- Docker Compose로 로컬 개발 환경을 구성한다.
- Swagger/OpenAPI로 API 문서를 제공한다.

## 6. 요청 유형 범위

MVP에서 우선 구현할 요청 유형은 아래 3개로 제한한다.

| 요청 유형 | 코드 | 설명 |
| --- | --- | --- |
| 비품/장비 요청 | ASSET_REQUEST | 노트북, 모니터, 키보드 등 사내 장비 요청 |
| 방문 신청 | VISITOR_REQUEST | 외부 방문자 출입 신청 |
| 시설 요청 | FACILITY_REQUEST | 회의실 장비 고장, 좌석 변경, 시설 점검 요청 |

### 6.1 요청 유형별 추가 입력값

요청 등록 화면은 공통 입력값과 요청 유형별 추가 입력값을 함께 사용한다.

공통 입력값:

- 요청 유형
- 제목
- 내용
- 우선순위

| 요청 유형 | 추가 입력값 |
| --- | --- |
| ASSET_REQUEST | 희망 자산 카테고리, 사용 시작일, 반납 예정일 |
| VISITOR_REQUEST | 방문자 이름, 방문자 연락처, 방문 일시, 방문 목적 |
| FACILITY_REQUEST | 위치, 문제 유형, 희망 처리일 |

아래 운영 요청 유형은 확장 기능으로 분류한다.

| 요청 유형 | 코드 | 처리 |
| --- | --- | --- |
| 구매 요청 | PURCHASE_REQUEST | MVP 이후 |
| 업무 지원 요청 | SUPPORT_REQUEST | MVP 이후 |

## 6.2 HR/전자결재 문서 유형 범위

근태와 전자결재는 운영 요청과 별도 도메인으로 관리하되, 결재 이력과 알림, 감사 이력은 공통 정책을 따른다.

| 문서/신청 유형 | 코드 | 처리 담당 |
| --- | --- | --- |
| 연차 신청서 | LEAVE_REQUEST | 팀장 1차 승인, HR 최종 확인 |
| 연차 취소 신청서 | LEAVE_CANCEL_REQUEST | 팀장 1차 승인, HR 최종 확인 |
| 재택근무 신청서 | REMOTE_WORK_REQUEST | 팀장 1차 승인, HR 최종 확인 |
| 외근 신청서 | OUT_OF_OFFICE_REQUEST | 팀장 1차 승인, HR 최종 확인 |
| 출장 신청서 | BUSINESS_TRIP_REQUEST | 팀장 1차 승인, HR 최종 확인 |
| 시차출퇴근 신청서 | STAGGERED_WORK_REQUEST | 팀장 1차 승인, HR 최종 확인 |
| 초과근무 신청서 | OVERTIME_REQUEST | 팀장 1차 승인, HR 최종 확인 |
| 지출결의서 | EXPENSE_REPORT | 팀장 1차 승인, 재무 최종 확인 |
| 법인카드 사용내역서 | CORPORATE_CARD_REPORT | 팀장 1차 승인, 재무 최종 확인 |
| 재직증명서 신청서 | EMPLOYMENT_CERTIFICATE | HR 처리 |
| 경력증명서 신청서 | CAREER_CERTIFICATE | HR 처리 |

## 7. 기능 요구사항

### 7.1 인증/사용자

- 사용자는 이메일, 비밀번호, 이름, 부서 정보를 입력해 회원가입할 수 있어야 한다.
- 이메일은 중복될 수 없다.
- 비밀번호는 서버에서 BCrypt로 암호화해 저장해야 한다.
- 사용자는 이메일과 비밀번호로 로그인할 수 있어야 한다.
- 로그인 성공 시 Access Token을 발급해야 한다.
- 로그인 사용자는 내 정보를 조회할 수 있어야 한다.
- 사용자는 로그아웃할 수 있어야 한다.
- 사용자는 MVP 이후 비밀번호를 변경할 수 있어야 한다.
- Refresh Token을 도입할 경우 사용자는 Access Token 만료 시 토큰을 재발급받을 수 있어야 한다.
- 비활성화된 사용자는 로그인할 수 없어야 한다.
- 관리자는 사용자 목록을 조회할 수 있어야 한다.
- 관리자는 사용자 상세 정보를 조회할 수 있어야 한다.
- 관리자는 필요한 경우 사용자의 권한을 변경할 수 있어야 한다.
- 관리자는 사용자를 비활성화하거나 다시 활성화할 수 있어야 한다.

### 7.2 권한

- 일반 직원은 본인이 생성한 요청과 예약만 수정/취소할 수 있어야 한다.
- 일반 직원은 팀장/운영 담당자/관리자 API에 접근할 수 없어야 한다.
- 팀장은 소속 팀원의 요청 승인 API에 접근할 수 있어야 한다.
- 운영 담당자는 최종 승인, 담당자 지정, 처리 상태 변경 API에 접근할 수 있어야 한다.
- HR 담당자는 연차/근태 정책, 근태 현황, 증명서 발급 요청, HR 관련 전자결재 문서에 접근할 수 있어야 한다.
- 재무 담당자는 지출결의서와 법인카드 사용내역서 관련 전자결재 문서에 접근할 수 있어야 한다.
- 관리자는 전체 요청, 예약, 자산, 사용자, 공통 코드, 감사 이력 정보를 조회할 수 있어야 한다.
- 권한 검증은 프론트엔드뿐 아니라 백엔드에서도 반드시 수행해야 한다.

### 7.3 요청

- 사용자는 요청 유형, 제목, 내용, 우선순위를 입력해 요청을 등록할 수 있어야 한다.
- 사용자는 요청 유형에 따라 필요한 추가 입력값을 입력할 수 있어야 한다.
- 사용자는 본인이 등록한 요청 목록을 조회할 수 있어야 한다.
- 사용자는 본인이 등록한 요청 상세를 조회할 수 있어야 한다.
- 사용자는 요청 상세 화면에서 수정 모드로 전환해 `PENDING_MANAGER_APPROVAL` 상태의 본인 요청만 수정할 수 있어야 한다.
- 사용자는 완료, 반려, 취소된 요청을 수정할 수 없어야 한다.
- 사용자는 완료 전 요청을 취소할 수 있어야 한다.
- 요청 목록은 상태, 유형, 우선순위, 날짜 기준으로 필터링할 수 있어야 한다.
- 요청 목록은 담당자, 마감일, 지연 여부, 미배정 여부 기준으로 필터링할 수 있어야 한다.
- 요청 목록은 페이지네이션과 정렬을 지원해야 한다.

### 7.4 요청 승인/처리

- 팀장은 소속 팀원의 요청을 1차 승인 또는 반려할 수 있어야 한다.
- 운영 담당자는 팀장 승인 완료 요청을 최종 승인 또는 반려할 수 있어야 한다.
- 운영 담당자는 최종 승인 시 담당자, 처리 예정일, 마감일을 지정할 수 있어야 한다.
- 운영 담당자 또는 관리자는 담당자를 변경할 수 있어야 한다.
- 운영 담당자 또는 요청 담당자는 승인된 요청을 처리 중 상태로 변경할 수 있어야 한다.
- 운영 담당자 또는 요청 담당자는 처리 중인 요청을 완료 상태로 변경할 수 있어야 한다.
- 반려 시 반려 사유를 입력할 수 있어야 한다.
- 요청 처리 과정에서 관리자 메모와 처리 코멘트를 입력할 수 있어야 한다.
- 모든 상태 변경은 이력으로 저장되어야 한다.
- 요청 승인 단계는 승인 이력으로 저장되어야 한다.
- 회의실 예약은 기본적으로 중복이 없으면 자동 승인한다.
- 특수 자원 예약은 운영 담당자 승인 대상으로 확장할 수 있다.

### 7.5 예약

- 사용자는 회의실, 날짜, 시작 시간, 종료 시간, 목적을 입력해 예약할 수 있어야 한다.
- 예약은 30분 단위로 등록할 수 있어야 한다.
- 예약 가능 시간은 평일 09:00부터 18:00까지로 제한한다.
- 같은 회의실에 시간이 겹치는 예약은 생성할 수 없어야 한다.
- 사용자는 본인의 예약 목록을 조회할 수 있어야 한다.
- 사용자는 예약 시작 전 본인 예약을 취소할 수 있어야 한다.
- 관리자는 전체 예약 목록을 조회하고 취소할 수 있어야 한다.
- 사용자는 예약 캘린더에서 날짜별/회의실별 예약 현황을 확인할 수 있어야 한다.
- 관리자는 예약 캘린더에서 전체 예약 현황을 확인할 수 있어야 한다.
- 관리자는 회의실/자원을 등록, 수정, 사용 가능 또는 사용 중지 상태로 변경할 수 있어야 한다.

### 7.6 자원

- 관리자는 회의실을 등록할 수 있어야 한다.
- 관리자는 회의실 이름, 위치, 수용 인원, 상태를 수정할 수 있어야 한다.
- 관리자는 회의실 상태를 사용 가능 또는 사용 중지로 변경할 수 있어야 한다.
- 사용 중지 상태의 회의실은 신규 예약 대상에서 제외되어야 한다.

### 7.7 자산

- 사용자는 자산 목록을 조회할 수 있어야 한다.
- 사용자는 본인이 대여 중이거나 과거 대여한 자산 목록을 조회할 수 있어야 한다.
- 관리자는 자산을 등록, 수정할 수 있어야 한다.
- 관리자는 자산 상태를 변경할 수 있어야 한다.

### 7.8 근태/연차

- 사용자는 본인의 연차 발생일수, 사용일수, 잔여일수를 조회할 수 있어야 한다.
- 사용자는 연차 발생, 사용, 취소, 관리자 조정 내역을 조회할 수 있어야 한다.
- 사용자는 연차, 반차, 시간차를 신청할 수 있어야 한다.
- 사용자는 승인 완료된 연차를 취소 신청할 수 있어야 한다.
- 연차 신청이 최종 승인되면 잔여 연차가 자동 차감되어야 한다.
- 연차 취소 신청이 최종 승인되면 차감된 연차가 복원되어야 한다.
- 사용자는 출근, 퇴근, 휴게, 근무시간 기록을 조회할 수 있어야 한다.
- 사용자는 재택근무, 외근, 출장, 시차출퇴근, 초과근무를 신청할 수 있어야 한다.
- 팀장은 소속 팀원의 근태/연차 신청을 승인 또는 반려할 수 있어야 한다.
- HR 담당자는 팀/부서별 근태 현황과 월간 근태 리포트를 조회할 수 있어야 한다.
- HR 담당자는 연차 발생 기준, 시간차 단위, 근무제 유형을 관리할 수 있어야 한다.

### 7.9 전자결재

- 사용자는 결재 문서를 작성, 임시저장, 상신, 취소할 수 있어야 한다.
- 사용자는 본인이 작성한 결재 문서 목록과 상세, 결재 이력을 조회할 수 있어야 한다.
- 결재 문서는 문서 유형별 입력값을 검증해야 한다.
- 팀장은 소속 팀원의 결재 문서를 승인 또는 반려할 수 있어야 한다.
- HR 담당자는 연차/근태/증명서 관련 문서를 최종 확인 또는 처리할 수 있어야 한다.
- 재무 담당자는 지출결의서와 법인카드 사용내역서를 최종 확인 또는 처리할 수 있어야 한다.
- 관리자는 전체 결재 문서와 결재 양식, 결재 상태를 관리할 수 있어야 한다.
- 결재선은 작성자, 팀장, 담당 부서 처리자로 구성하며 문서 유형에 따라 담당 부서가 달라져야 한다.
- 결재 단계별 승인자, 승인 시각, 반려 사유, 처리 코멘트는 이력으로 저장되어야 한다.
- 결재 상태 변경, 증명서 발급 처리, 비용 결재 처리는 감사 이력 대상이어야 한다.
- 자산 코드는 중복될 수 없어야 한다.
- 점검 중 또는 폐기 상태의 자산은 대여할 수 없어야 한다.
- 자산 대여 시 상태는 `IN_USE`로 변경되어야 한다.
- 자산 반납 시 상태는 `AVAILABLE`로 변경되어야 한다.
- 자산 대여와 반납은 이력으로 저장되어야 한다.

### 7.8 대시보드

- 관리자는 전체 요청 수를 확인할 수 있어야 한다.
- 관리자는 요청 상태별 건수를 확인할 수 있어야 한다.
- 관리자는 월별 요청 건수를 확인할 수 있어야 한다.
- 관리자는 자산 상태별 현황을 확인할 수 있어야 한다.
- 관리자는 당일 또는 주간 예약 현황을 확인할 수 있어야 한다.
- 관리자는 지연 요청 수를 확인할 수 있어야 한다.
- 관리자는 미배정 요청 수를 확인할 수 있어야 한다.
- 관리자는 오늘 처리 예정 요청 수를 확인할 수 있어야 한다.

### 7.9 공통 코드/마스터 데이터

- 시스템은 부서, 요청 유형, 우선순위, 자산 카테고리, 시설 문제 유형, 자원 유형, 사용자 상태, 요청 상태 기준을 정의해야 한다.
- MVP에서는 DB seed 또는 enum으로 공통 코드를 관리한다.
- 공통 코드 관리 화면은 MVP 이후로 미룰 수 있다.
- 프론트엔드 select 입력값은 공통 코드 기준을 따른다.

### 7.10 감사 이력

- 관리자의 주요 작업은 감사 이력으로 저장되어야 한다.
- 감사 이력에는 작업자, 작업 유형, 대상 리소스, 변경 전/후 값, 작업 시간이 포함되어야 한다.
- MVP 필수 감사 대상은 요청 승인/반려, 담당자 지정/변경, 사용자 비활성화, 자원 사용 중지다.
- 2순위 감사 대상은 자산 상태 변경, 사용자 권한 변경이다.

### 7.11 공통 화면

- 존재하지 않는 URL로 접근하면 404 Not Found 화면을 표시해야 한다.
- 권한이 없는 화면으로 접근하면 권한 없음 화면을 표시해야 한다.
- 비로그인 사용자가 인증이 필요한 화면에 접근하면 로그인 화면으로 이동해야 한다.

## 11. 성공 기준

- 일반 직원과 관리자가 서로 다른 화면과 권한으로 서비스를 사용할 수 있다.
- 일반 직원은 요청과 예약을 등록하고 상태를 확인할 수 있다.
- 팀장은 소속 팀원의 요청을 1차 승인 또는 반려할 수 있다.
- 운영 담당자는 요청을 최종 승인하고 담당자와 마감일을 지정할 수 있다.
- 운영 담당자 또는 요청 담당자는 요청을 처리 중, 완료 상태로 변경할 수 있다.
- 일반 직원은 근태와 전자결재 문서를 작성하고 진행 상태를 확인할 수 있다.
- 팀장은 소속 팀원의 근태/전자결재 문서를 1차 승인 또는 반려할 수 있다.
- HR 담당자는 연차, 근태, 증명서 문서를 최종 확인하고 처리할 수 있다.
- 재무 담당자는 지출결의서와 법인카드 사용내역서를 최종 확인하고 처리할 수 있다.
- 연차 신청 승인/취소 승인에 따라 연차 잔여일수가 자동으로 변경된다.
- 출퇴근 기록과 근무제 신청 내역을 근태 리포트에서 확인할 수 있다.
- 요청 상태 변경 이력이 누락 없이 저장된다.
- 관리자 주요 작업 감사 이력이 저장된다.
- 같은 회의실에 겹치는 시간 예약이 생성되지 않는다.
- 자산 대여/반납 시 자산 상태와 이력이 함께 변경된다.
- 관리자가 회의실/자원을 등록하고 예약 가능 여부를 제어할 수 있다.
- 주요 목록 화면은 검색, 필터, 페이지네이션을 지원한다.
- 존재하지 않는 URL 접근 시 404 화면이 표시된다.
- Swagger 문서로 API를 확인하고 테스트할 수 있다.
