# OfficeOps Hub 기능 명세서

## 문서 버전 이력

| 버전 | 기준 | 수정 사항 | 삭제 사항 |
| --- | --- | --- | --- |
| v1.0.0 | 관리자 전용 계정 생성 반영 이전 기능 명세 | 공개 회원가입과 기존 사용자 관리 기능 기준선 | 없음 |
| v1.1.0 | 관리자 전용 계정 생성 반영 | `AUTH-001`을 관리자 사용자 계정 생성으로 변경, `USER-001` 사용자 계정 생성 기능 추가, 사용자 목록/상세/권한 변경 번호 재정렬, 생성 가능 역할에 HR/재무 포함 | 비로그인 사용자의 공개 회원가입 처리 규칙 제거 |
| v1.2.0 | 문서 네이밍 및 버전 관리 체계 정리 | 문서 파일명을 번호 없는 한글 제목 기반 규칙으로 정리하고 버전 표기를 semantic version 형식으로 통일 | 문서 번호 접두어와 영문 기반 산출물 파일명 제거 |
| v1.3.0 | 파일명 버전 최신화 규칙 반영 | 문서 파일명의 버전을 문서 내부 최신 버전과 동일하게 관리하도록 정리하고, 이후 수정 및 버전 상승 시 파일명과 참조 링크를 즉시 갱신하는 규칙 추가 | 최신 버전과 맞지 않는 파일명 버전 표기 제거 |

## 1. 문서 목적

이 문서는 `OfficeOps Hub`의 주요 기능별 동작 방식, 입력값, 처리 규칙, 예외 조건을 정의한다.

백엔드 API 구현, 프론트엔드 화면 구현, 테스트 케이스 작성의 기준 문서로 사용한다.

## 3. 인증 기능

### AUTH-001. 관리자 사용자 계정 생성

| 항목 | 내용 |
| --- | --- |
| 설명 | 시스템 관리자가 사내 사용자 계정을 생성한다. |
| 접근 권한 | ROLE_ADMIN |
| 입력값 | 이메일, 이름, 부서, 초기 권한, 계정 상태, 임시 비밀번호 |
| 결과 | 사용자 계정 생성 |

처리 규칙:

- 이메일은 필수값이다.
- 이메일은 중복될 수 없다.
- 임시 비밀번호 또는 초기 비밀번호는 필수값이다.
- 비밀번호는 서버에서 BCrypt로 암호화해 저장한다.
- 기본 권한은 `ROLE_USER`이며, `ROLE_ADMIN`이 필요한 업무 역할을 지정할 수 있다.
- 운영 담당자, HR 담당자, 재무 담당자는 사용자 계정을 생성할 수 없다.
- 사용자 생성 이력은 감사 이력으로 저장한다.

예외:

| 조건 | 처리 |
| --- | --- |
| 이메일 중복 | 사용자 생성 실패 |
| 필수값 누락 | Validation 에러 |
| 이메일 형식 오류 | Validation 에러 |

### AUTH-002. 로그인

| 항목 | 내용 |
| --- | --- |
| 설명 | 이메일과 비밀번호로 로그인한다. |
| 접근 권한 | 비로그인 사용자 |
| 입력값 | 이메일, 비밀번호 |
| 결과 | Access Token, 사용자 정보 반환 |

처리 규칙:

- 이메일과 비밀번호가 일치하면 Access Token을 발급한다.
- 사용자 상태가 `INACTIVE`이면 로그인할 수 없다.
- 비밀번호 검증은 BCrypt 매칭으로 처리한다.
- 응답에는 사용자 id, 이름, 이메일, 부서, 권한을 포함한다.
- Refresh Token은 MVP 이후 고도화할 수 있다.

예외:

| 조건 | 처리 |
| --- | --- |
| 존재하지 않는 이메일 | 로그인 실패 |
| 비밀번호 불일치 | 로그인 실패 |
| 비활성화 사용자 | 로그인 실패 |
| 필수값 누락 | Validation 에러 |

### AUTH-003. 로그아웃

| 항목 | 내용 |
| --- | --- |
| 설명 | 로그인 상태를 해제한다. |
| 접근 권한 | 로그인 사용자 |
| 입력값 | 없음 |
| 결과 | 클라이언트 인증 상태 초기화 |

처리 규칙:

- Access Token만 사용하는 MVP에서는 프론트엔드에서 토큰과 사용자 상태를 제거한다.
- Refresh Token을 사용할 경우 서버에서 Refresh Token을 무효화한다.

### AUTH-005. 비밀번호 변경

| 항목 | 내용 |
| --- | --- |
| 설명 | 로그인 사용자가 본인 비밀번호를 변경한다. |
| 접근 권한 | 로그인 사용자 |
| 우선순위 | MVP 이후 |
| 입력값 | 현재 비밀번호, 새 비밀번호, 새 비밀번호 확인 |

처리 규칙:

- 현재 비밀번호가 일치해야 한다.
- 새 비밀번호와 새 비밀번호 확인이 일치해야 한다.
- 새 비밀번호는 BCrypt로 암호화해 저장한다.
- 비밀번호 변경 후 기존 토큰을 유지할지 재로그인시킬지는 구현 시 결정한다.

### AUTH-006. Refresh Token 재발급

| 항목 | 내용 |
| --- | --- |
| 설명 | Access Token 만료 시 Refresh Token으로 새 Access Token을 발급한다. |
| 접근 권한 | Refresh Token 보유 사용자 |
| 우선순위 | MVP 이후 |
| 입력값 | Refresh Token |

처리 규칙:

- Refresh Token은 HttpOnly Cookie 저장을 권장한다.
- Refresh Token이 유효하면 새 Access Token을 발급한다.
- Refresh Token이 만료되었거나 유효하지 않으면 재로그인이 필요하다.

### AUTH-007. 사용자 비활성화/활성화

| 항목 | 내용 |
| --- | --- |
| 설명 | 관리자가 사용자 계정을 비활성화하거나 다시 활성화한다. |
| 접근 권한 | ROLE_ADMIN |
| 우선순위 | 필수 |

사용자 상태:

- `ACTIVE`
- `INACTIVE`

처리 규칙:

- 비활성화 사용자는 로그인할 수 없다.
- 로그인 성공 시 마지막 로그인 시각을 갱신할 수 있다.
- 비활성화해도 기존 요청, 예약, 이력 데이터는 삭제하지 않는다.
- 자기 자신의 계정을 비활성화하는 기능은 제한할 수 있다.
- 사용자 비활성화/활성화 작업은 감사 이력으로 저장한다.

## 4. 요청 기능

### REQ-001. 요청 등록

| 항목 | 내용 |
| --- | --- |
| 설명 | 일반 직원이 사내 요청을 등록한다. |
| 접근 권한 | 로그인 사용자 |
| 입력값 | 요청 유형, 제목, 내용, 우선순위 |
| 초기 상태 | PENDING_MANAGER_APPROVAL |

요청 유형:

- `ASSET_REQUEST`
- `VISITOR_REQUEST`
- `FACILITY_REQUEST`

우선순위:

- `LOW`
- `NORMAL`
- `HIGH`
- `URGENT`

처리 규칙:

- 요청 등록자는 현재 로그인 사용자로 설정한다.
- 요청 생성 시 상태는 `PENDING_MANAGER_APPROVAL`로 저장한다.
- 제목과 내용은 필수값이다.
- 요청 유형에 따라 추가 입력값을 검증한다.
- 요청 생성 시 최초 이력도 함께 저장할 수 있다.
- 요청자의 팀장 정보를 기준으로 1차 승인 대상자를 결정한다.

예외:

| 조건 | 처리 |
| --- | --- |
| 요청 유형 누락 | Validation 에러 |
| 제목 누락 | Validation 에러 |
| 내용 누락 | Validation 에러 |
| 허용되지 않은 요청 유형 | Validation 에러 |

조회 정보:

- 요청자 정보

### REQ-004. 요청 수정

| 항목 | 내용 |
| --- | --- |
| 설명 | 사용자가 본인 요청을 수정한다. |
| 접근 권한 | 요청 작성자 |
| 수정 가능 상태 | PENDING_MANAGER_APPROVAL |
| 수정 방식 | 요청 상세 화면 내 수정 모드 |
| 수정 가능 항목 | 제목, 내용, 우선순위, 요청 유형별 추가 입력값 |

처리 규칙:

- 본인이 작성한 요청만 수정할 수 있다.
- `PENDING_MANAGER_APPROVAL` 상태에서만 수정 가능하다.
- 관리자 처리 이후 상태에서는 수정할 수 없다.
- 별도 수정 페이지를 만들지 않고 요청 상세 화면에서 수정 모드로 전환한다.

예외:

| 조건 | 처리 |
| --- | --- |
| 다른 사용자의 요청 수정 | 403 |

| 완료/반려/취소 요청 수정 | 상태 변경 불가 에러 |

### REQ-005. 요청 취소

유형별 추가 입력값:

- 빠른 구현을 우선하면 유형별 추가 정보를 JSON 컬럼 또는 별도 `request_details` 구조로 저장할 수 있다.
- API 명세 작성 단계에서 최종 저장 방식을 확정한다.

## 7. 예약 기능

### RSV-001. 회의실 예약 등록

| 항목 | 내용 |
| --- | --- |
| 설명 | 사용자가 회의실을 예약한다. |
| 접근 권한 | 로그인 사용자 |
| 입력값 | 회의실 id, 시작 시간, 종료 시간, 목적 |
| 초기 상태 | RESERVED |

처리 규칙:

- 예약 시작 시간은 종료 시간보다 이전이어야 한다.
- 예약은 30분 단위로만 가능하다.
- 예약 가능 시간은 평일 09:00부터 18:00까지다.
- 같은 회의실에 시간이 겹치는 예약이 있으면 등록할 수 없다.
- 취소된 예약은 중복 검사 대상에서 제외한다.

예약 충돌 조건:

```text
새 예약 시작 시간 < 기존 예약 종료 시간
AND
새 예약 종료 시간 > 기존 예약 시작 시간
```

예외:

| 조건 | 처리 |
| --- | --- |
| 시간이 겹치는 예약 존재 | 예약 실패 |
| 시작 시간이 종료 시간보다 늦음 | Validation 에러 |
| 예약 가능 시간 외 입력 | Validation 에러 |
| 존재하지 않는 회의실 | 404 |

## 18. 공통 처리 규칙

- 모든 목록 조회는 페이지네이션을 적용한다.
- 모든 생성/수정 API는 입력값 검증을 수행한다.
- 권한이 없는 요청은 403 응답을 반환한다.
- 존재하지 않는 리소스는 404 응답을 반환한다.
- 잘못된 상태 변경은 상태 전이 에러를 반환한다.
- 요청 상태 변경은 반드시 이력 테이블에 저장한다.
- 요청 승인은 반드시 승인 이력 테이블에 저장한다.
- 관리자 주요 작업은 감사 이력으로 저장한다.
- 마스터 데이터는 문서화된 공통 코드 기준을 따른다.
