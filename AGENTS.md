# Agent Roles

이 문서는 `OfficeOps Hub` 저장소에서 AI 에이전트 또는 자동화 도구가 작업할 때 따라야 하는 프로젝트 지침이다.

에이전트는 이 저장소를 단일 회사 내부 운영 시스템을 만드는 팀 프로젝트로 이해하고, 문서와 구현 결과가 같은 방향을 유지하도록 작업한다.

## 0. Agent 요청 관련 지침

- Agent는 프로젝트 작업을 돕기 위한 보조 수단일 뿐이며, 프로젝트 산출물의 일부로 드러나면 안 된다.
- `AGENTS.md`를 제외한 프로젝트의 모든 문서에는 agent와의 작업 내용, 요청 내용, 응답 내용, 작업 주체 표현을 언급하지 않는다.
- 프로젝트 문서에서는 agent, 에이전트, AI 도구 작업, 자동화 도구 요청 같은 표현을 사용하지 않는다.
- 단, 시스템 기능으로서의 `User-Agent`처럼 기술 표준 용어는 예외로 한다.
- 커밋 메시지, PR description, README, 기획서, 명세서, 테스트 문서에는 agent의 개입 사실을 남기지 않는다.
- agent 작업 규칙이 필요하면 반드시 이 문서에만 기록한다.

## 1. 프로젝트 이해

- 프로젝트명은 `OfficeOps Hub`이다.
- 프로젝트 성격은 사내 운영, HR, 전자결재, 요청, 예약, 자산 관리를 통합하는 웹 애플리케이션이다.
- 대상 사용자는 일반 직원, 팀장, 운영 담당자, HR 담당자, 재무 담당자, 시스템 관리자이다.
- 1차 구현은 일반 사용자 기능과 시스템 완성에 필요한 관리자/담당자 기능을 포함한다.
- 2차 이후에는 추가 기능과 고도화를 진행한다.
- 공개 회원가입은 사용하지 않고, 시스템 관리자가 사용자 계정을 생성한다.
- HR 기능은 실제 문서 파일 기반 처리가 아니라 시스템 내 요청, 승인, 일정 확인 중심으로 구현한다.

## 3. README 작성 원칙

README는 팀 프로젝트 기준으로 작성한다.

- 첫 화면에서 프로젝트 목적, 대상 사용자, 핵심 기능이 보여야 한다.
- 팀 프로젝트임을 명시한다.
- 기술 스택을 영역별로 구분해 작성한다.
- 로컬 실행 방법과 환경 변수를 새 팀원이 따라 할 수 있게 작성한다.
- 문서 링크는 최신 파일명을 기준으로 연결한다.
- 협업 규칙, 브랜치 전략, PR 규칙은 팀 운영 문서와 충돌하지 않아야 한다.
- 라이선스는 `LICENSE` 파일 기준으로 안내한다.

README를 수정할 때는 이 문서의 팀 프로젝트 기준과 `README.md`의 기존 구성을 함께 따른다.

## 4. 문서 버전 관리

- 문서별 독립 버전 관리 방식을 사용한다.
- 각각의 문서 작업은 기본 `v1.0.0`으로 시작한다.
- 문서마다 최신 버전은 달라도 된다.
- 실제로 수정한 문서만 버전을 상승시킨다.
- 수정한 문서는 파일명 버전과 문서 내부 최신 버전을 일치시킨다.
- 파일명 변경이 발생하면 해당 파일을 참조하는 `README.md`와 관련 문서 링크만 함께 수정한다.
- 버전 이력에는 수정 사항과 삭제 사항을 모두 기록한다.
- 삭제된 화면, API, 기능, 정책도 삭제 사항에 명시한다.
- 단순 오탈자나 링크 수정도 문서 산출물에 영향을 주면 해당 문서 버전을 갱신한다.
- 프로젝트 전체 버전은 `0.1.0`으로 시작한다.
- 실제 기능 구현 완료 시 프로젝트 전체 버전을 `1.0.0`으로 올린다.
- minor 변경점은 `0.1.0` 단위로, major 변경점은 `1.0.0` 단위로 올린다.
- 기능 추가, 기능 수정, 문서 작업 전에는 관련 버전 이력을 확인하고 필요하면 즉시 버전 업한다.

## 5. 산출물 네이밍

프로젝트 관련 문서는 아래 형식을 따른다.

```text
OfficeOps_hub_[문서제목]_v[major].[minor].[patch].[확장자]
```

작성 규칙:

- 문서 번호 접두어를 붙이지 않는다.
- 문서 제목은 한글을 기본으로 한다.
- 루트 `README.md`는 GitHub 대표 문서이므로 파일명을 유지한다.
- 파일명 버전은 해당 문서 내부 최신 버전과 맞춘다.

## 6. 기능 범위 기준

1차 구현에서 일반 사용자 대상 기능은 다음 범위를 기준으로 한다.

- 로그인/인증
- 지원 요청
- 비품 요청
- HR 기본 기능 요청
- 회의실, 라운지 등 공간 예약
- 내 요청과 예약 상태 확인
- 시스템 내 전자결재 기반 승인 흐름 확인

시스템 완성을 위해 함께 고려할 관리자/담당자 기능:

- 시스템 관리자 사용자 계정 생성
- 역할과 권한 관리
- 요청 승인/반려/처리
- 예약과 자산 관리
- 감사 이력 조회
- 대시보드

## 7. 권한 기준

- `ROLE_ADMIN`은 시스템 관리자이다.
- 시스템 관리자는 사용자 계정 생성, 역할 부여, 전체 운영 현황, 감사 이력을 관리한다.
- 일반 관리자는 별도 단일 역할로 뭉뚱그리지 않고 업무 역할에 따라 구분한다.
- 주요 업무 역할은 팀장, 운영 담당자, HR 담당자, 재무 담당자이다.
- 일반 사용자는 본인 요청, 본인 예약, 본인 전자결재 관련 화면을 중심으로 사용한다.

권한이나 역할이 바뀌면 요구사항, 기능 명세, 화면 정의, API 명세, 상태/권한 정책을 함께 확인한다.

## 8. 화면 흐름과 프로토타입

- 기존 외부 HTML 흐름을 그대로 복사하지 않는다.
- 현재 프로젝트에서 구현할 화면 전체 흐름을 기준으로 작성한다.
- 화면 흐름도는 페이지 간 이동 관계가 한눈에 보여야 한다.
- 카드나 화면 노드는 간결하게 유지하되, 제목과 핵심 기능을 알아볼 수 있어야 한다.
- 프로토타입을 수정하면 화면 흐름도 문서와 README 링크를 확인한다.

## 9. Git과 원격 업로드

- `main`은 배포 가능한 안정 상태로 유지한다.
- 모든 작업 브랜치는 로컬과 원격 모두 `dev` 하위 브랜치로 생성한다.
- PR 대상 브랜치는 항상 `dev`로 고정한다.
- 기능 구현은 `feature/feature_name` 형식의 브랜치를 생성한다.
- 기능 브랜치에서는 해당 기능만 구현한다.
- 새로운 기능 구현은 새 기능 브랜치로 이동해서 진행한다.
- 문서 작업은 `docs/docs_name` 형식의 브랜치를 생성한다.
- 문서 브랜치에서는 해당 문서 작업만 진행한다.
- 기능과 문서 외 작업은 작업 성격에 맞는 이름으로 브랜치를 생성한다.
- 원격 업로드 요청 시 commit, push, PR까지 진행한다.
- 커밋 메시지는 `[YYYY.MM.DD hwan] type work_name` 형식으로 작성한다.
- 현재 작업 환경의 주인은 `hwan`으로 고정하며, 커밋 주체를 `agent`로 작성하지 않는다.
- 작업 환경이 바뀌면 커밋 전 작업 환경의 주인을 묻고 확인한다.
- PR description은 `작업 핵심 내용`, `기존과의 변경점` 두 항목으로 간단하게 작성한다.
- PR에는 작업 내용, 테스트 결과, 문서 변경 여부를 포함한다.
- PR merge 전 테스트를 통과해야 한다.
- 민감 정보는 커밋하지 않는다.
- `.env`는 커밋하지 않고 `.env.example` 또는 `.env.sample`만 공유한다.
- 문서 파일명 변경은 링크 갱신과 함께 커밋한다.

## 10. 코드와 문서 동기화

에이전트는 코드 또는 문서를 수정할 때 다음을 확인한다.

- 기능 변경이 요구사항과 기능 명세에 반영되어 있는가?
- API 변경이 인터페이스 명세에 반영되어 있는가?
- 화면 변경이 화면 정의와 화면 흐름도에 반영되어 있는가?
- 상태나 권한 변경이 상태/권한 정책에 반영되어 있는가?
- 테스트 기준이 테스트 계획서와 맞는가?
- README 링크가 실제 파일을 가리키는가?

## 11. 금지 사항

- 사용자 변경 사항을 임의로 되돌리지 않는다.
- 문서 번호 접두어를 다시 추가하지 않는다.
- 공개 회원가입을 기본 흐름으로 되살리지 않는다.
- 비밀값, 토큰, 운영 DB 비밀번호, JWT Secret을 커밋하지 않는다.
- README와 동일한 내용을 목적 없이 중복 문서로 만들지 않는다.
- 문서 파일명만 바꾸고 내부 버전 이력이나 참조 링크를 방치하지 않는다.

## 12. 작업 완료 전 확인

작업 완료 전에는 다음을 확인한다.

- 변경된 문서의 버전 이력이 갱신되었는가?
- 변경된 문서의 파일명 버전과 내부 최신 버전이 일치하는가?
- README의 문서 링크가 끊기지 않았는가?
- `.gitignore`에 로컬 환경, IDE, 빌드 산출물, 비밀 파일이 포함되어 있는가?
- Git 상태에서 의도하지 않은 파일이 포함되지 않았는가?
